# M365 Copilot Chat API 接入说明（含 Tenant ID / Client ID / Client Secret / SharePoint 配置）

更新时间：2026-03-12


## 1. 接口形态与核心结论

Microsoft 365 Copilot Chat API（Graph beta）不是单一 `/chat` 端点，而是两步：

1. `POST /beta/copilot/conversations` 创建会话，拿到 `conversationId`
2. `POST /beta/copilot/conversations/{conversationId}/chat` 或 `/chatOverStream` 继续对话

该 API 仍为 beta，官方说明不建议用于生产环境，且要求调用用户具备 Copilot 许可。


## 2. 前提条件（必须满足）

1. 账号与许可
   - 工作或学校账号（不支持个人 Microsoft 账号）
   - 用户拥有 Microsoft 365 Copilot 许可（并具备对应基础订阅）

2. 权限类型
   - 仅支持 Delegated（委派权限）
   - 不支持 Application（应用权限）

3. Microsoft Graph 委派权限（缺一不可）
   - `Sites.Read.All`
   - `Mail.Read`
   - `People.Read.All`
   - `OnlineMeetingTranscript.Read.All`
   - `Chat.Read`
   - `ChannelMessage.Read.All`
   - `ExternalItem.Read.All`


## 3. 如何获取 Application (client) ID、Directory (tenant) ID、Client Secret

下面按微软官方门户实际操作路径写，适合首次配置。

### 3.1 进入正确租户

1. 打开 `https://entra.microsoft.com`
2. 登录后，右上角切换到你要接入的目标租户
3. 进入 `Identity` -> `Applications` -> `App registrations`

如果你在多个租户中操作，先确认右上角目录名称是否正确，再创建应用。

### 3.2 新建应用注册（若已有可跳过）

1. 在 `App registrations` 页面点击 `New registration`
2. 填写：
   - `Name`：例如 `m3652api-local`
   - `Supported account types`：建议选择 `Accounts in this organizational directory only`
3. 点击 `Register`

完成后会进入该应用的 `Overview` 页面。

### 3.3 获取 Directory (tenant) ID 与 Application (client) ID

在应用 `Overview` 页面可直接看到：

- `Application (client) ID`：这就是 `client_id`
- `Directory (tenant) ID`：这就是 `tenant_id`

可直接复制使用。

### 3.4 创建 Client Secret（仅在需要机密客户端时）

1. 进入应用左侧 `Certificates & secrets`
2. 在 `Client secrets` 标签页点击 `New client secret`
3. 填写：
   - `Description`：例如 `m3652api-dev-secret`
   - `Expires`：建议按公司安全策略选择最短可接受周期
4. 点击 `Add`
5. 立刻复制 `Value` 列的密钥值并安全保存

重要：

- 你离开页面后无法再次看到完整 secret 值，只能重新新建
- 建议把 secret 放入密码管理器或 Key Vault，不要写入代码仓库

### 3.5 给应用添加 Graph 委派权限并管理员同意

1. 进入 `API permissions` -> `Add a permission`
2. 选择 `Microsoft Graph` -> `Delegated permissions`
3. 添加第 2 节列出的全部权限
4. 回到 `API permissions` 页面点击 `Grant admin consent for <TenantName>`
5. 确认每个权限状态为已授予（Granted）

### 3.6 配置 Redirect URI（授权码登录必需）

由于 Copilot Chat API 只支持 **Delegated**，本项目需要走 **OAuth 2.0 授权码（Authorization Code）** 获取 `refresh_token`，以便服务端后续自动刷新访问令牌。

你需要在 Entra ID 应用中配置 Redirect URI：

1. 进入应用左侧 `Authentication`
2. 点击 `Add a platform`（添加平台）
3. 选择 `Web`
4. 在 `Redirect URIs` 中添加（按你的端口调整）：
   - `http://localhost:8217/m365/oauth/callback`
5. 保存（Save）

说明：

- 这是“本机单用户代理”形态下的推荐配置
- 如果你修改了 `config.yaml` 的端口或希望自定义回调地址，请同时修改 Redirect URI 与配置中的 `m365.redirect-uri`


## 4. 本项目（m3652api）实际需要哪些参数

本仓库当前实现改为静态凭据模式：服务启动时从 `config.yaml` 读取 `m365` 段。

最小必填如下：

- `m365.tenant-id`
- `m365.client-id`
- `m365.client-secret`
- `m365.timezone`（建议显式配置，示例：`Pacific/Auckland`）
- `m365.redirect-uri`（建议显式配置并与 Entra Redirect URI 保持一致）
- 如果需要图片能力，还需要：
  - `m365.image-upload.enabled`
  - `m365.image-upload.target`
  - `m365.image-upload.sharepoint-hostname`
  - `m365.image-upload.sharepoint-site-path`

示例：

```yaml
m365:
  auth-id: "m365-static"
  tenant-id: "<TENANT_ID>"
  client-id: "<CLIENT_ID>"
  client-secret: "<CLIENT_SECRET>"
  timezone: "Pacific/Auckland"
  # App-only scopes（仅作为兜底；Copilot Chat API 仍要求 delegated token）
  scopes:
    - "https://graph.microsoft.com/.default"
  # Delegated scopes（授权码登录使用，用于获取 refresh_token）
  delegated-scopes:
    - "openid"
    - "profile"
    - "offline_access"
    - "https://graph.microsoft.com/Files.ReadWrite.AppFolder"
    - "https://graph.microsoft.com/User.Read"
    - "https://graph.microsoft.com/Sites.Read.All"
    - "https://graph.microsoft.com/Mail.Read"
    - "https://graph.microsoft.com/People.Read.All"
    - "https://graph.microsoft.com/OnlineMeetingTranscript.Read.All"
    - "https://graph.microsoft.com/Chat.Read"
    - "https://graph.microsoft.com/ChannelMessage.Read.All"
    - "https://graph.microsoft.com/ExternalItem.Read.All"
  redirect-uri: "http://localhost:8217/m365/oauth/callback"
  image-upload:
    enabled: true
    target: "sharepoint"
    sharepoint-hostname: "contoso.sharepoint.com"
    sharepoint-site-path: "/sites/Engineering"
```

### 4.1 首次授权（浏览器登录，不在命令行交互）

启动服务后，用浏览器打开：

- `http://localhost:8217/m365/oauth/start`

完成登录后，可查看：

- `http://localhost:8217/m365/oauth/status`

当 `has_refresh_token=true` 时，说明服务端已经保存了可用于持续刷新的委派凭据。

### 4.2 上游连通性自检（可选）

如果你想在不写任何 curl 的情况下快速确认：

1. 委派 token 是否可用（`GET /v1.0/me`）
2. Copilot Chat API 是否可创建会话（`POST /beta/copilot/conversations`）

可以在浏览器打开：

- `http://localhost:8217/m365/upstream/check`

说明：

- 该接口会创建一个 Copilot conversation（用于验证权限与许可），不会发送聊天内容
- 如只想验证委派 token，可访问：`http://localhost:8217/m365/upstream/check?copilot=0`


### 4.3 图片上传到 SharePoint 的配置说明

当前项目的图片链路不是把 `input_image` 原始字节直接发给 Copilot Chat API，而是先上传到 SharePoint，再把上传结果的 `webUrl` 写入 `contextualResources.files[].uri`。

项目当前行为如下：

1. 只接受 OpenAI Responses 风格的 `data:image/...;base64,...`
2. 解析图片并上传到指定 SharePoint site 默认 drive 的 `special/approot`
3. 取上传文件的 `driveItem.webUrl`
4. 注入到 `chatOverStream.contextualResources.files[]`
5. 请求结束后最佳努力删除这批临时文件

因此，图片能力要想可用，除了 Copilot Chat 的委派权限外，还必须额外满足：

1. delegated scopes 中包含 `https://graph.microsoft.com/Files.ReadWrite.AppFolder`
2. `m365.image-upload.enabled=true`
3. `m365.image-upload.target=sharepoint`
4. 正确填写 `sharepoint-hostname` 与 `sharepoint-site-path`
5. 完成一次新的浏览器授权，让 refresh token 带上新 scope

如果你是在已有环境上追加图片能力，改完配置后必须重新访问：

- `http://localhost:8217/m365/oauth/start`

否则即使 `config.yaml` 已写好，新 scope 也不会自动出现在现有 refresh token 上。


### 4.4 如何准备 SharePoint 站点

建议准备一个专门用于代理上传的站点，例如：

- 团队站点：`https://contoso.sharepoint.com/sites/Engineering`
- 团队站点：`https://contoso.sharepoint.com/teams/PlatformOps`
- 多地理区域站点：`https://contosoeur.sharepoint.com/sites/EngineeringEU`

建议做法：

1. 选择一个业务上可以接受临时文件上传的站点
2. 确保发起 Copilot 请求的用户自己能正常打开这个站点
3. 确保该站点有默认文档库
4. 不需要手动创建 `Apps` 文件夹或应用子目录

说明：

- 根据 Microsoft Graph 的 app folder 文档，第一次访问 `special/approot` 时，系统会在目标 drive 下自动创建应用目录。
- app folder 文件会占用 SharePoint 站点配额，因此不建议把它当长期大文件存储。
- 本项目当前实现写入的是站点默认 drive 的 `approot`，不是你手工指定的某个文档库路径。


### 4.5 如何获取 `sharepoint-hostname`

`sharepoint-hostname` 就是 SharePoint 站点 URL 里的主机名部分：

- 不包含 `https://`
- 不包含后面的路径
- 不包含末尾的 `/`

最直接的获取方式就是看站点 URL。

示例 1：

- 站点 URL：`https://contoso.sharepoint.com/sites/Engineering`
- `sharepoint-hostname`：`contoso.sharepoint.com`
- `sharepoint-site-path`：`/sites/Engineering`

示例 2：

- 站点 URL：`https://contoso.sharepoint.com/teams/PlatformOps`
- `sharepoint-hostname`：`contoso.sharepoint.com`
- `sharepoint-site-path`：`/teams/PlatformOps`

示例 3（多地理区域）：

- 站点 URL：`https://contosoeur.sharepoint.com/sites/EngineeringEU`
- `sharepoint-hostname`：`contosoeur.sharepoint.com`
- `sharepoint-site-path`：`/sites/EngineeringEU`

常见错误写法：

- 错误：`https://contoso.sharepoint.com`
- 正确：`contoso.sharepoint.com`

- 错误：`contoso.sharepoint.com/sites/Engineering`
- 正确：`contoso.sharepoint.com`

- 错误：`contoso.sharepoint.com/`
- 正确：`contoso.sharepoint.com`


### 4.6 如何获取 `sharepoint-site-path`

`sharepoint-site-path` 是站点 URL 中主机名后面的 server-relative path。

规则：

1. 必须以 `/` 开头
2. 必须是站点路径，不是文档库、文件夹或文件路径
3. 当前项目建议填写具体站点，例如 `/sites/Engineering` 或 `/teams/PlatformOps`
4. 不要把文档库路径拼进去
5. 当前实现不建议填写 SharePoint 根站点 `/`，请优先使用明确的子站点路径

示例：

- 站点 URL：`https://contoso.sharepoint.com/sites/Engineering`
- `sharepoint-site-path`：`/sites/Engineering`

- 站点 URL：`https://contoso.sharepoint.com/teams/PlatformOps`
- `sharepoint-site-path`：`/teams/PlatformOps`

错误示例：

- 错误：`/sites/Engineering/Shared Documents`
- 原因：这已经是文档库路径，不是 site path

- 错误：`Shared Documents`
- 原因：缺少前导 `/`，而且这也不是 site path

- 错误：`/sites/Engineering/Forms/AllItems.aspx`
- 原因：这是页面路径，不是 site path


### 4.7 从浏览器和 SharePoint 管理后台获取站点地址

如果你不确定站点 URL，可以这样找：

方法 1：直接从浏览器地址栏获取

1. 用目标用户登录 Microsoft 365
2. 打开目标 SharePoint 站点
3. 复制地址栏中的完整站点 URL
4. 按上一节拆成：
   - `sharepoint-hostname`
   - `sharepoint-site-path`

方法 2：从 SharePoint Admin Center 获取

1. 打开 `https://admin.microsoft.com`
2. 进入 SharePoint 管理中心
3. 打开 `Active sites`
4. 找到你准备使用的站点
5. 查看该站点的 URL
6. 按 URL 拆出 hostname 与 site path

如果你在多地理区域租户中工作，不同站点可能使用不同 hostname，例如：

- `contoso.sharepoint.com`
- `contosoeur.sharepoint.com`
- `contosoapc.sharepoint.com`

这种情况下，不要假设所有站点都共用同一个 hostname，必须以目标站点真实 URL 为准。


### 4.8 用 Microsoft Graph 验证 SharePoint 配置是否正确

拿到 `sharepoint-hostname` 和 `sharepoint-site-path` 后，可以先用 Graph 验证目标 site 是否能被正确解析。

示例：

```bash
curl -X GET "https://graph.microsoft.com/v1.0/sites/contoso.sharepoint.com:/sites/Engineering" \
  -H "Authorization: Bearer <ACCESS_TOKEN>" \
  -H "Accept: application/json"
```

如果配置正确，你会拿到一个 `site` 对象，其中包含：

- `id`
- `name`
- `webUrl`
- `displayName`

接着可以验证默认 drive 的 `approot` 是否可访问：

```bash
curl -X GET "https://graph.microsoft.com/v1.0/sites/<SITE_ID>/drive/special/approot" \
  -H "Authorization: Bearer <ACCESS_TOKEN>" \
  -H "Accept: application/json"
```

如果这个请求成功，说明至少下面几件事成立：

1. 站点路径能被正确解析
2. 站点默认 drive 可用
3. `Files.ReadWrite.AppFolder` 权限链路基本正常

说明：

- `approot` 大小写敏感，必须写成小写
- 第一次访问时，Microsoft Graph 可能会自动创建该 app folder

### 4.10 如何在服务端确认配置是否生效

启动服务后访问：

- `http://localhost:8217/m365/oauth/status`

重点看以下字段：

- `image_upload_enabled`
  - `true` 说明服务端已启用图片上传功能
- `image_upload_target`
  - 应为 `sharepoint`
- `sharepoint_site_configured`
  - `true` 说明 hostname 和 site path 至少已填写
- `image_upload_required_scope`
  - 应为 `https://graph.microsoft.com/Files.ReadWrite.AppFolder`
- `image_upload_scope_ready`
  - `true` 说明当前 token 已具备图片上传所需 scope

如果 `image_upload_enabled=true` 但 `image_upload_scope_ready=false`，通常说明：

1. 你修改了 delegated scopes，但还没重新走 `/m365/oauth/start`
2. 租户管理员还没对 `Files.ReadWrite.AppFolder` 完成 consent
3. 当前登录用户没有拿到新的 refresh token


### 4.11 常见 SharePoint 配置错误

1. 把 `sharepoint-hostname` 写成完整 URL
   - 错误：`https://contoso.sharepoint.com`
   - 正确：`contoso.sharepoint.com`

2. 把文档库路径写进 `sharepoint-site-path`
   - 错误：`/sites/Engineering/Shared Documents`
   - 正确：`/sites/Engineering`

3. 改了 delegated scopes，但没有重新授权
   - 结果：`image_upload_scope_ready=false`，上传报 403

4. 使用了错误的站点 hostname
   - 多地理区域环境下尤其常见
   - 应以目标站点浏览器 URL 为准，不要用想当然的主域名代替

5. 站点本身不可访问
   - 目标用户在浏览器里都打不开该站点时，Graph 上传通常也会失败

6. 误以为必须手工创建 `Apps/<AppName>`
   - 不需要，`approot` 路径第一次访问时会自动创建


## 5. 请求调用最小示例

### 5.1 创建会话

```bash
curl -X POST "https://graph.microsoft.com/beta/copilot/conversations" \
  -H "Authorization: Bearer <ACCESS_TOKEN>" \
  -H "Content-Type: application/json" \
  -d "{}"
```

### 5.2 继续聊天（同步）

```bash
curl -X POST "https://graph.microsoft.com/beta/copilot/conversations/<CONVERSATION_ID>/chat" \
  -H "Authorization: Bearer <ACCESS_TOKEN>" \
  -H "Content-Type: application/json" \
  -d '{
    "message": { "text": "Summarize my latest unread emails." },
    "locationHint": { "timeZone": "Pacific/Auckland" }
  }'
```

### 5.3 继续聊天（流式 SSE）

```bash
curl -N -X POST "https://graph.microsoft.com/beta/copilot/conversations/<CONVERSATION_ID>/chatOverStream" \
  -H "Authorization: Bearer <ACCESS_TOKEN>" \
  -H "Content-Type: application/json" \
  -H "Accept: text/event-stream" \
  -d '{
    "message": { "text": "Give me a concise action plan." },
    "locationHint": { "timeZone": "Pacific/Auckland" }
  }'
```


## 6. 常见问题排查

1. 找不到 `tenant_id` 或 `client_id`
   - 你不在应用 `Overview` 页面，或租户切错了

2. `client secret` 创建后复制不到
   - 只会显示一次，必须创建后立即复制；丢失只能重建

3. 调用报权限不足（403）
   - 通常是权限没全加，或者没点击 `Grant admin consent`

4. 调用报不支持权限类型
   - 你用了 application token；该 API 只支持 delegated

5. 时间相关回答不准确
   - 缺少 `locationHint.timeZone`，或时区不是 IANA 格式
